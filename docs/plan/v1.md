# LLD (v1) — Phases 1–3 (functionality only)

This document is the low-level design for implementing the benchmark through Phase 3:
- Phase 1: deterministic preparation + validation + scoring (no agent automation).
- Phase 2: semi-automated manual runs (VSCode + Kilo agent optional).
- Phase 3: batch runs via Kilo CLI (`--auto --json`) on host (still non-secure).

Security hardening (Docker isolation + egress allowlist) is Phase 6 and out of scope here.

## 0) Guiding constraints

- Do not commit competition data, private labels, submissions, Kilo transcripts, DBs, or run workspaces.
- Runs must be reproducible “enough”: deterministic splits + pinned versions later; some ML nondeterminism is acceptable.
- Per-competition leaderboard only (no cross-task aggregation yet).

## 2) Competition contract

Each competition is defined by `competitions/<competition_id>/spec.yaml` plus a preparation script.

### 2.1 `spec.yaml` (minimum)

```yaml
id: <competition_id>
task_type: regression   # regression|binary|multiclass

target_column: <label_col_name>
id_column: <id_col_name>   # if missing in raw, prepare step must create one and record it

metric:
  name: rmse               # rmse|mae|logloss|auc
  higher_is_better: false

submission:
  filename: submission.csv
  prediction_column: <pred_col_name>   # usually equals target_column

split:
  strategy: random         # random|stratified|group|time
  test_size: 0.2
  seed: 1337
  group_column: null
  time_column: null

budgets:
  time_seconds: 900

prepare:
  raw_train_path: raw/train.csv
  raw_test_path: raw/test.csv
  raw_sample_submission_path: raw/sample_submission.csv
```

### 2.2 Generated outputs (prepare step)

`prepare_competition.py` produces:

Public (agent-visible; copied into run workspace):
- `competitions/<id>/public/train_public.csv`
- `competitions/<id>/public/test_public.csv`
- `competitions/<id>/public/sample_submission.csv`
- `competitions/<id>/public/README_task.md`
- `competitions/<id>/public/public_manifest.json`

Private (never given to agent; used only for scoring):
- `competitions/<id>/private/holdout_labels.parquet` (must include `id_column` + `target_column`)
- `competitions/<id>/private/split.json` (seed + strategy + split mapping file hash)
- `competitions/<id>/private/split_mapping.csv` (`id_column`, `split` = `train_public`|`holdout`)

`public_manifest.json` (minimum):
```json
{
  "competition_id": "...",
  "created_at": "...",
  "spec_hash": "sha256:...",
  "files": {
    "train_public.csv": {"sha256": "...", "rows": 123, "cols": 45},
    "test_public.csv": {"sha256": "...", "rows": 456, "cols": 44},
    "sample_submission.csv": {"sha256": "...", "rows": 456, "cols": 2},
    "README_task.md": {"sha256": "..."}
  }
}
```

## 3) Validator contract (`orchestrator/validate.py`)

Input:
- competition `spec.yaml`
- public files: `sample_submission.csv`, `test_public.csv`
- candidate `submission.csv`

Validation rules (minimum):
- file exists and is readable as CSV
- columns match `sample_submission.csv` exactly (same names; order not important)
- rowcount matches `test_public.csv`
- `id_column` values: no missing, no duplicates
- covers the same ID set as `test_public.csv` (by set equality)
- prediction column is numeric-coercible; NaNs are rejected by default

Output:
- structured result:
  - `ok: bool`
  - `status: success|invalid_submission|runtime_error|...`
  - `errors: [{code, message}]`
  - `normalized_submission_path` (optional): path to a rewritten, canonicalized submission (IDs ordered to match `test_public.csv`)

Design note: accept arbitrary row order and canonicalize by joining on `id_column`. This reduces “dumb failures”.

## 4) Scorer contract (`orchestrator/score.py`)

Input:
- competition `spec.yaml`
- private holdout labels (`id_column`, `target_column`)
- normalized submission (`id_column`, `prediction_column`)

Behavior:
- merge by `id_column`
- compute `score_raw` using `metric.name`
- compute `score_normalized` so leaderboard is always higher-is-better:
  - if `higher_is_better: false`, `score_normalized = -score_raw`
  - else `score_normalized = score_raw`

Output:
- `score_raw`, `score_normalized`, `metric_name`, plus any warnings (e.g., clipping applied)

## 5) Run lifecycle (Phases 2–3)

### 5.1 Run directory

`runs/<run_id>/` (all ignored by git):
- `workspace/` — agent-visible workspace (contains public data + agent code edits)
- `artifacts/`
  - `submission.csv` (final; copied from workspace)
  - `kilo.json` / `kilo.log` (captured transcript; filename TBD by implementation)
  - `stdout.txt`, `stderr.txt`
- `result.json` — run metadata + outcome (also ignored by git; canonical record goes into DB)

### 5.2 Run identity

`run_id` should be derived from:
- `competition_id`
- provider/model config (provider, model_id, temperature, etc.)
- seeds (split seed from spec + run seed)
- prompt hash (base + override + rendered params)
- `public_manifest.json` hash
- scaffold version (Kilo version + settings hash)

Implementation can start with a short random ID plus storing all the above in `result.json`; later switch to content-derived IDs.

## 6) Phase 2 execution mode (manual)

Goal: repeatable manual runs in VSCode without attempting security.

Workflow:
- Create `runs/<run_id>/workspace/` with:
  - `public/` copied from `competitions/<id>/public/`
  - an empty `work/` for outputs
  - a `RUN_INSTRUCTIONS.md` (rendered from prompt template; tells the agent exactly what to do)
- Open `runs/<run_id>/workspace/` in VSCode and run Kilo agent manually.
- After run: copy `submission.csv` to `runs/<run_id>/artifacts/`, validate+score on host, write `result.json`, append to DB.

## 7) Phase 3 execution mode (batch via Kilo CLI)

Goal: orchestrator drives Kilo CLI, captures artifacts, and produces results without VSCode.

`orchestrator/run_one.py` responsibilities:
- create run directory + workspace template
- render prompt = `prompts/base_prompt.md` + optional `prompts/competition_overrides/<id>.md`
- start Phase 3 with a short CLI capability spike to verify the Kilo CLI invocation and available artifacts (do not assume `--auto --json` exists)
- call Kilo CLI headlessly in whatever supported mode exists (exact invocation is expected to be discovered during the spike)
- enforce wall-clock timeout (kill process after `budgets.time_seconds` + small grace)
- collect artifacts (submission + transcript + stdout/stderr)
- validate → score → write `result.json` → insert DB

`orchestrator/sweep.py` responsibilities:
- iterate models × seeds × competitions
- rate-limit / backoff on transient API errors
- emit progress summaries and a final leaderboard build

## 8) Results store (SQLite, Phase 2+)

DB file: `results/results.sqlite` (ignored by git).

Minimum tables:
- `runs`:
  - `run_id` (pk)
  - `competition_id`
  - `provider`, `model_id`, `temperature`, `max_tokens` (nullable)
  - `split_seed`, `run_seed`
  - `status`
  - `score_raw`, `score_normalized`
  - `runtime_seconds`
  - `created_at`
  - `public_manifest_sha256`, `prompt_sha256`, `kilo_version`, `benchmark_version`
  - `artifact_dir` (filesystem pointer)

Leaderboard builder reads from DB and can write optional artifacts (not tracked by default):
- `python -m orchestrator.leaderboard --import-results --write-root`

Repo-root summary is maintained in `results.md`; legacy leaderboards are archived under `archive/leaderboards/`.

## 9) Testing (Phase 1)

Add fast tests for:
- split determinism (same raw + spec → same split.json + holdout labels)
- validator correctness on small fixtures
- scoring correctness on small fixtures

Keep tests unit-level and <2 minutes total.
