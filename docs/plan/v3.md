# Phase 3+ plan (v3 onwards)

This document describes what should happen after Phase 2 (`v2`) and how to approach Phases 3–6 without relying on unverified assumptions.

## Phase 3 (v3) — Batch execution on host (no Docker)

### Primary goal
Replace the manual VSCode UI loop with a headless/batch harness that can run many agent attempts consistently and produce a model-centric leaderboard.

### First task: Kilo CLI capability spike (non-negotiable)
Before building automation, verify what Kilo can actually do from the command line:
- Can we run headlessly (no VSCode) given a workspace directory?
- Can we provide the prompt deterministically (file/arg)?
- Can we capture a transcript / structured output (JSON or logs)?
- What exit codes/errors look like for common failure modes?
- Can we enforce a hard wall-clock timeout by killing the Kilo process and still finalize cleanly?

**Outputs of the spike:**
- A short doc snippet in `docs/plan/v3.md` (this file) with the exact CLI invocation(s) that work.
- A list of artifacts available per run (stdout/stderr, transcript file, etc.).
- A go/no-go decision:
  - **Go:** implement an `auto` command in `orchestrator/run_one.py`.
  - **No-go:** revise Phase 3 to an alternative automation approach (see “Fallbacks” below).

### Implementation plan (if CLI is workable)
1) Add `python -m orchestrator.run_one auto ...`:
   - creates run workspace (like Phase 2)
   - starts timer
   - invokes Kilo CLI with a rendered prompt
   - enforces wall-clock timeout (kill process)
   - collects artifacts (transcript/logs/stdout/stderr if available)
   - validates → scores → records (leaderboard writing is optional)
2) Add `python -m orchestrator.sweep ...`:
   - loops over models × repetitions (and later seeds)
   - rate limits and retries transient provider failures
   - writes progress summaries and a final leaderboard refresh
3) Make model identity required for automation:
   - require `--provider` and `--model-id` for `auto`/`sweep`
   - keep `mode/temperature/max_tokens` optional but recorded if provided

### Fallbacks (if CLI is not workable)
Phase 3 should be redefined explicitly before building a large sweep system. Candidate fallback options:
- Extend Phase 2: keep manual VSCode runs but standardize metadata capture and make sweeps “human-assisted”.
- Build an alternate headless harness (only if it can run the same “agent” behavior outside VSCode/Kilo UI).

## Phase 4 (v4) — Reproducibility packaging + baselines

### Goal
Reduce run-to-run drift and improve auditability (still not secure).

### Concrete deliverables
- Pin tool versions:
  - Python dependencies (lockfile)
  - Kilo version (and any relevant settings/config snapshots)
- Expand provenance:
  - prompt hash (rendered)
  - spec hash + public data manifest hashes
  - run config fields (provider/model_id/mode/temp/max_tokens)
- Add per-competition non-agent baselines (host-side) for sanity checks.

## Phase 5 (v5) — Multi-competition benchmark + publishable leaderboard artifact

### Goal
Run a model sweep across ~5 competitions and publish model-centric outputs.

### Concrete deliverables
- Add 4 more competitions under `competitions/` with documented selection rationale.
- One command produces:
  - per-competition “best-by-model” summaries (see `results.md` for the committed snapshot)
  - an auditable run log (DB-backed)
- Add `REPRODUCIBILITY.md` with rerun instructions and pinned versions.

Notes:
- Cross-task “overall leaderboard” definition is deferred (per PRD notes). Phase 5 can still output per-competition model summaries.

### Phase 5.5 (v5.5) — Expand models + 5th competition + replication

Goal: reduce result noise (variance) and broaden coverage before starting Phase 6 security work.

Deliverables:
- Add a 5-competition suite file (5 real competitions; keep `toy_regression` as fixtures only).
- Expand model sets (ideally a “main / tool-capable” set and an “experimental” set).
- Run replicated sweeps (`--runs-per-model 3–5`) into a dedicated sqlite DB path and refresh `results.md`.

## Phase 6 (v6) — Security hardening

### Goal
Enforce integrity constraints so results are security-relevant.

### Concrete deliverables
- Isolation (Docker or equivalent) with strict mounts:
  - `public/` read-only
  - `work/` read-write
  - `private/` not mounted
- Egress allowlist (deny all outbound except model API endpoints).
- Disable side channels (MCP disabled; no extra tools).
- Resource limits and explicit “egress/mount” tests.

## Versioning reminders
- Do not start `v(N+1)` without explicit human approval.
- Commit often; include agent number in commit messages.
